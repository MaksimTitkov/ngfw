# NGFW Manager — Changelog

---

## v2.2 — Cross-SU Sync Edition

> Синхронизация объектов и правил между системами управления, зеркальные правила, аккордеон блоков, проверка подсетей.

---

### Новые возможности

#### Cross-SU Object Sync — синхронизация объектов между СУ

Новая вкладка **Objects** на странице **Policy Diff** (`/diff`):

- Выбери две или более СУ — система сравнивает их объекты (сети, группы, сервисы) и показывает расхождения
- Одна кнопка синхронизирует объект из источника на целевую СУ с сохранением структуры и членов группы
- Результат немедленно попадает в локальный кэш — без ожидания ресинхронизации

#### Rule Sync — массовое копирование правил между СУ

Новая вкладка **Rule Sync** на странице **Policy Diff**:

- Сравнивает политики двух устройств (в рамках одной СУ или разных)
- Показывает правила, которые есть на источнике, но отсутствуют на целевом устройстве
- Bulk-копирование отмеченных правил одним действием, включая автоматическое создание всех нужных объектов
- Если на целевом устройстве нет папок — предлагает создать их автоматически

#### Reverse Rule — зеркальное правило

Из контекстного меню правила (или bulk-выбора) доступна команда **Reverse**:

- Создаёт копию правила с обменом `sourceAddr` ↔ `destAddr` и `sourceZone` ↔ `destZone`
- Новое правило получает суффикс `_rev` и создаётся в состоянии **disabled**
- Если `_rev` уже существует — добавляется числовой суффикс (`_rev_2`, `_rev_3`)
- Поддерживается для нескольких правил одновременно

#### Умный анализ конфликтов при переносе и реверсе

При Transfer и Reverse система анализирует конфликты имён:

| Ситуация | Поведение |
|----------|-----------|
| Правило идентично существующему | Пропускается — дубль не создаётся |
| Имя занято, содержимое отличается | Добавляется суффикс `_copy` / `_rev_N`, предупреждение |
| Правило уникально | Создаётся в обычном режиме |

#### Экспорт правил в JSON (raw формат СУ)

Новый пункт в разделе **Export** — выгрузка правил устройства в JSON-формате, идентичном формату API PT NGFW, с группировкой по секциям `pre / default / post`.

#### Аккордеон блоков в таблице правил

Блоки внутри папки теперь сворачиваются и разворачиваются интерактивно:

- Заголовок показывает имя блока, подсеть и количество правил
- Состояние сохраняется в `sessionStorage` — перезагрузка не сбрасывает раскладку
- Секция **General** (правила без блока) выделена отдельным заголовком в конце списка

#### Создание правила прямо в блоке

Кнопка **+** у каждого заголовка блока открывает форму создания правила с уже выбранным блоком — без ручного выбора блока в форме.

#### Проверка подсети при добавлении правила в блок

Если объекты источника или назначения не попадают в подсеть блока — появляется предупреждение с тремя вариантами действия: перенести в рекомендованный блок / добавить в текущий / добавить в General. Разрешение объектов рекурсивное (до 5 уровней вложенности групп).

## v2.1 — IP Sorter Edition

> Новый интеллектуальный модуль сортировки правил по IP-плану. Полный CRUD IP-плана с шифрованием данных.

---

### Новые возможности

#### IP Sorter — автоматическая классификация правил по IP-плану

Новая страница **Sorter** (`/sort-analysis`) анализирует правила безопасности и классифицирует их по блокам на основе IP-плана устройства.

- Сопоставление `sourceAddr` и `destAddr` правила с подсетями из IP-плана
- Группировка правил по VRF → VLAN → subnet
- Предложение структуры папок и блоков — без автоматического применения
- Fallback: если источник не совпадает с планом — анализируется назначение

#### IP-план на устройство

Каждое устройство имеет **собственный независимый IP-план**, хранимый в БД:

- Страница **System → IP Plan** — управление планом для выбранного устройства
- Полный CRUD: добавить строку, редактировать inline, удалить строку, очистить всё
- Загрузка из файла: формат A (`vrf, vlan, subnet, [mask], [description]`) и автоопределение формата B
- **Шифрование**: все поля (VRF, VLAN, subnet, description) хранятся в зашифрованном виде через Fernet (AES-128-CBC, ключ из `SECRET_KEY`)

#### Блоки внутри папок

Правила с назначенным блоком (`block_id`) теперь отображаются в таблице правил с визуальным разделителем — синяя полоска слева с именем блока и подсетью.

#### Scheduled Sort Analysis

В планировщике задач (`System → Scheduled Tasks`) добавлен тип задачи **Sort Analysis** — автоматический запуск анализа по расписанию с логированием результатов.

---

### Изменения безопасности

| Компонент | Было | Стало |

## v2.0 — Multi-SU Edition

> Крупный релиз. Полный переход на мультисистемную архитектуру.

---

### Главные изменения

#### Поддержка нескольких систем управления (Multi-SU)

Раньше NGFW Manager работал с одной системой управления (СУ). Теперь в одном экземпляре приложения можно подключить несколько независимых СУ и работать с каждой из них в рамках единого интерфейса.

- Таблица `management_systems` — реестр всех подключённых СУ с хостом, учётными данными и именем
- Все объекты, правила и устройства привязаны к конкретной СУ через `su_id`
- Синхронизация, создание папок, deploy — работают независимо для каждой СУ
- Страница «Objects» получила двухуровневый селектор: **СУ → устройство** — исключены коллизии при одинаковых именах устройств в разных СУ (типичная ситуация для клонированных prod/test-систем)

#### Умный перенос правил между устройствами (Cross-SU Transfer)

Ключевая функция версии 2.0 — перенос правил из одной СУ в другую с автоматическим разрешением объектов.

| Сценарий | Поведение |
|----------|-----------|
| Одна СУ, то же устройство | Точное совпадение по UUID — объекты не дублируются |
| Одна СУ, глобальный объект | Проверка наличия UUID на целевом устройстве — повторное создание не требуется |
| Разные СУ | Поиск совпадения по fingerprint (структуре объекта), затем по имени; при отсутствии — создание копии |
| Глобальный объект → другая СУ | Автоматическое создание device-специфичной копии с немедленной записью в локальный кэш |

После переноса все созданные объекты сразу сохраняются в локальный кэш (`CachedObject`) — новое правило отображается корректно без ожидания следующей синхронизации.

#### Scoped ID — изоляция объектов между СУ

Все внешние идентификаторы (UUID объектов, устройств, правил) хранятся в формате `{su_id}:{raw_uuid}`. Это полностью исключает коллизии при работе с несколькими СУ, имеющими пересекающиеся UUID-пространства (например, клонированные системы).

#### Vue.js SPA Frontend

Фронтенд переведён на Vue.js (Vite). Исходный код собирается на этапе Docker-сборки (`Stage 1: frontend-builder`) и встраивается в статику Python-приложения — более отзывчивый интерфейс без перезагрузки страниц.

#### Предкомпилированные модули

Четыре ключевых модуля поставляются в виде скомпилированных нативных расширений Python (`.so`): клиент NGFW API, движок синхронизации, движок переноса правил и анализатор политик. Это исключает необходимость дополнительной сборки при деплое и обеспечивает стабильную работу в любом совместимом окружении.

---

### Новые возможности интерфейса

#### Страница Objects — двухуровневый селектор

В v1.x все устройства всех СУ отображались плоским списком вкладок. При наличии нескольких СУ с одинаковыми именами устройств было невозможно понять, объекты какой именно системы просматриваются.

В v2.0 добавлен двухуровневый селектор:
- **Строка 1** — вкладки СУ (хост системы управления)
- **Строка 2** — вкладки устройств, принадлежащих выбранной СУ

URL сохраняет выбор: `/objects?su_id=X&device_id=D&type_filter=networks`

#### Post-Transfer Cache

После переноса правила все объекты, созданные на целевом устройстве, немедленно видны в менеджере с правильными именами — без ожидания полной ресинхронизации.

---

### Архитектурные изменения

| Компонент | v1.x | v2.0 |
|-----------|------|------|
| Поддержка СУ | Одна | Несколько (`management_systems`) |
| Хранение ID объектов | Чистый UUID | `{su_id}:{uuid}` (scoped) |
| Перенос правил | В рамках одной СУ | Между любыми СУ и устройствами |
| Разрешение объектов при переносе | По имени / UUID | Fingerprint → имя → создание |
| Кэш после создания объекта | При следующей синхронизации | Немедленно (`_cache_new_object`) |
| Frontend | Server-side Jinja2 | Vue.js SPA (Vite, собирается в Docker) |
| Ключевые модули | Исходный код | Предкомпилированные `.so` (нет необходимости в сборке) |
| Источник `su_id` при переносе | Сессия пользователя | Папка правила (авторитетный источник) |

## v1.1 (stable) — предыдущий релиз

> Стабильная production-версия.

### Новые разделы и страницы

| Раздел | URL | Описание |
|--------|-----|----------|
| Dashboard | `/dashboard` | Сводная страница: счётчики правил/объектов, статус устройств, список последних изменений, алерты, быстрая навигация |
| Smart Search | `/search` | Поиск по IP, CIDR, порту, имени правила. Режимы: Any / Name / IP / Port. Поддержка Ctrl+K overlay на всех страницах |
| Policy Analyzer | `/analyzer` | Локальный анализ кэша правил: теневые, слишком широкие, избыточные, отключённые правила |
| Change Log | `/changelog` | Журнал всех изменений: кто/что/когда. Фильтры по дате, пользователю, типу, действию. Постраничная загрузка |
| Policy Diff | `/diff` | Сравнение политик двух device-групп; просмотр локально изменённых правил (pending deploy) |
| Rule Templates | `/templates` | Библиотека шаблонов: сохранение правила как шаблона, применение в любую папку любого устройства, bulk-удаление |

### Управление объектами (полный CRUD)

- Создание сетевых объектов — Host/Network, CIDR, FQDN, IP-range
- Редактирование объектов через Edit-модалку
- Удаление объектов — одиночное и bulk
- Сетевые группы — Create / Update / Delete членов группы
- Сервисы и группы сервисов — полный CRUD (TCP, UDP, ICMP)
- Object Usage Map — поиск всех правил, где используется объект

### Правила безопасности

- Bulk-операции: enable/disable, action, log, move to folder, delete, save as template
- Экспорт в Excel — цветовая кодировка action, auto-filter, имена объектов
- Экспорт структуры папок в YAML

### NAT

- Редактирование NAT-правила через Edit-модалку
